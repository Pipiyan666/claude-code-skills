## CC 7大核心技巧

### 1. 固定项目文件夹
**规则：** 每次 CC 开始时，先 `cd` 到项目文件夹。
**Why:** 建立持久的项目记忆体系，让 AI 记住项目上下文。
**How to apply:** 第一步总是切换到项目目录。

### 2. 记录错误与教训
**规则：** 每次犯错，不时要求 CC 把经验教训写到记忆里。
**Why:** 避免重复犯同样的错误，持续优化工作方式。
**How to apply:** 当出现意外结果或错误时，立即要求更新记忆。

### 3. Git 工作流（头号重点）
**规则：** 
- 安装 GitHub CLI
- 任何一次修改都必须 `commit + push`
- 写入记忆和 CLAUDE.md，作为头号重点

**Why:** 保证代码可追溯性、团队协作、进度记录。
**How to apply:** 每次代码变更完成后，立即 commit/push，更新 CLAUDE.md 和内存。

### 4. 详细日志
**规则：** 任何一个模块都要求最详细打点日志，放在 `./logs` 文件夹。
**Why:** 便于问题排查、性能分析、业务追踪。
**How to apply:** 代码中合理配置日志级别，日志输出到项目的 logs/ 目录。

### 5. 测试优先
**规则：** 每一次修改前，要求 CC 提前设计测试 case。
**Why:** 确保功能正确性、减少回归、提高代码质量。
**How to apply:** 单元测试、模块测试、集成测试全覆盖，自动化优先。

### 6. 无情的迭代测试
**规则：** 不厌其烦地去迭代 AI，无情地测试机器。一天要配合 TA 测试几十次。
**Why:** 通过大量测试驱动优化，发现边界情况和改进空间。
**How to apply:** 尽量自动化，不能自动化就靠人工；频繁测试，记录每次测试结果。

### 7. 核心架构掌控
**规则：** 
- PM 自己掌控核心技术架构，尤其是 AI 相关的
- 严格要求 AI 按照之前讨论好的架构来做
- 整个项目共用一个 repo
- 可使用 worktree，本地开多个分支，每个分支代表一种技术架构
- 代码设计一定不要相互影响
- 仍然保持每次修改必须 `commit + push` 到相应分支

**Why:** 确保架构一致性、便于架构对比、减少重复工作。
**How to apply:** 
  - 设计时明确架构版本
  - 使用 worktree 隔离不同架构的代码
  - 定期审视架构决策
  - 所有分支都保持提交历史清晰
